# fix: Add generic 404 redirect

## Summary

Add a real `/404` route, route unmatched URLs to it with history replacement, and redirect missing Project / Project Task views there instead of showing inconsistent inline missing-resource states.

---

## Problem Frame

Users can land on stale or invalid Kanai URLs. The origin requirements define one consistent recovery path for missing Projects, missing Project Tasks, and unmatched routes.

---

## Requirements

- R1. Provide a generic 404 page at `/404`.
- R2. Missing Project and Project Task pages redirect to the generic 404 page.
- R3. Unknown or unmatched app URLs end up on the generic 404 page.
- R4. The generic 404 page includes a primary “Go to Projects” action to `/`.
- R5. Redirects to `/404` replace browser history to avoid back-button loops.

**Origin actors:** A1 User
**Origin flows:** F1 Missing project recovery, F2 Missing task recovery, F3 Unknown route recovery
**Origin acceptance examples:** AE1 Missing project redirects, AE2 Missing task redirects, AE3 Unknown route redirects

---

## Scope Boundaries

- Do not update `CONTEXT.md`; this is routing/UI behavior, not domain language.
- Do not create an ADR unless implementation uncovers a hard-to-reverse architectural decision.
- Do not redesign general error handling.
- Do not add custom per-resource 404 copy.
- Do not hand-edit `client/src/routeTree.gen.ts`; regenerate it through the frontend build.

## Context & Research

### Relevant Code and Patterns

- `client/src/routes/` contains TanStack Router file routes; new routes should live there.
- `client/src/router.tsx` creates the router and is the likely place for a global unmatched-route redirect.
- `client/src/routes/__root.tsx` renders the app shell through `AppShell`.
- `client/src/domains/workspace/ui/ProjectBoardPage.tsx` loads the Project through `api.projects.get(projectId)` and currently shows a generic load error on project query failure.
- `client/src/domains/workspace/ui/TaskDetailPage.tsx` derives the selected task from the loaded project task list and currently shows inline “This task could not be found.” UI.
- `client/src/api/openapi-client/runtime.ts` defines `ResponseError`, which exposes the underlying `response.status` for generated API failures.
- `client/src/domains/workspace/ui/templates/WorkspaceLayout.tsx` provides the existing workspace shell used by pages.

### Institutional Learnings

- No relevant `docs/solutions/` learning exists for frontend routing, TanStack Router 404 handling, or missing-resource redirects.

### External References

- None. Existing TanStack Router usage in the repo is sufficient for this small routing fix.

---

## Key Technical Decisions

- Use a real `/404` route: keeps the canonical destination bookmarkable and prevents the unmatched-route handler from being the page itself.
- Redirect unmatched routes through router-level not-found handling: covers unknown URLs without adding route-specific catch-all pages.
- Detect missing Project via generated API `ResponseError` status: redirects only true not-found responses and preserves existing auth/network/server error behavior.
- Detect missing Project Task from the loaded task list: matches current page behavior while changing the outcome from inline fallback to redirect.
- Avoid a shared not-found abstraction for now: two call sites do not justify a new helper unless implementation reveals duplication that meaningfully improves clarity.

---

## Open Questions

### Resolved During Planning

- Should `/404` be a real route or only a not-found component? Use a real route so redirects have a canonical destination.

### Deferred to Implementation

- Exact unmatched-route hook placement: prefer router-level not-found redirect with replacement; use a file-based catch-all redirect route only if the installed TanStack Router behavior cannot change the URL to `/404` from router configuration.

## Implementation Units

### U1. Add canonical 404 page and unmatched-route redirect

**Goal:** Create the generic 404 page and ensure unknown URLs land at `/404` with history replacement.

**Requirements:** R1, R3, R4, R5; F3; AE3

**Dependencies:** None

**Files:**
- Create: `client/src/routes/404.tsx`
- Modify: `client/src/router.tsx`
- Generated by build: `client/src/routeTree.gen.ts`
- Test: `client/src/routes/404.test.tsx`
- Test: `client/src/router.test.tsx` or nearest existing router test location if one already exists during implementation

**Approach:**
- Add a `/404` file route that renders inside the normal workspace shell.
- Keep copy generic and recovery-focused.
- Add a primary “Go to Projects” link to `/`.
- Configure unmatched-route handling to navigate to `/404` with replacement.
- Keep `/404` itself as a normal route so it does not trigger the not-found redirect loop.

**Patterns to follow:**
- `client/src/routes/index.tsx` for route-file shape.
- `client/src/domains/workspace/ui/templates/WorkspaceLayout.tsx` for page shell.
- `client/src/router.tsx` for router configuration.

**Test scenarios:**
- Happy path: rendering the 404 page shows generic not-found messaging and a “Go to Projects” link targeting `/`.
- Integration: an unmatched URL causes router-level navigation to `/404` instead of rendering an empty or fallback shell.
- Edge case: `/404` itself renders the page and does not redirect again.
- Integration: unmatched-route navigation uses replacement semantics so browser history does not preserve the invalid URL as the immediate back target.

**Verification:**
- A fresh build regenerates route types and includes the `/404` route.
- Unknown app paths land on `/404` and expose the Projects recovery action.

### U2. Redirect missing Project and Project Task views

**Goal:** Replace missing-resource inline states with canonical `/404` redirects while preserving non-404 error handling.

**Requirements:** R1, R2, R4, R5; F1, F2; AE1, AE2

**Dependencies:** U1

**Files:**
- Modify: `client/src/domains/workspace/ui/ProjectBoardPage.tsx`
- Modify: `client/src/domains/workspace/ui/TaskDetailPage.tsx`
- Test: `client/src/domains/workspace/ui/ProjectBoardPage.test.tsx`
- Test: `client/src/domains/workspace/ui/TaskDetailPage.test.tsx`

**Approach:**
- In the Project board, redirect only when the project query failed because the generated API returned a true not-found response.
- Leave existing auth, network, and server-error UI in place.
- In Task detail, redirect when the Project request or task-list request fails with a true not-found response.
- Also redirect when task loading has completed successfully and the selected task ID is absent from the project task list.
- Use replacement navigation for all missing-resource redirects.

**Patterns to follow:**
- Existing query/error branches in `client/src/domains/workspace/ui/ProjectBoardPage.tsx`.
- Existing missing-task branch in `client/src/domains/workspace/ui/TaskDetailPage.tsx`.
- Existing router mocks in nearby page tests.

**Test scenarios:**
- Covers AE1. Given the Project request returns a not-found response, rendering the Project board redirects to `/404` with replacement.
- Error path: given the Project request fails with a non-404 error, the existing “Project details could not be loaded.” UI remains visible and no `/404` redirect occurs.
- Covers AE2. Given tasks load successfully but the route task ID is absent, rendering Task detail redirects to `/404` with replacement.
- Covers AE2. Given the task route's Project request or task-list request returns a true not-found response, rendering Task detail redirects to `/404` with replacement.
- Error path: given the task list request fails with a non-404 error, the existing retryable “Task details could not be loaded.” UI remains visible and no missing-task redirect occurs while the failure is unresolved.
- Edge case: while tasks are still pending, Task detail does not redirect prematurely.

**Verification:**
- Missing Project and missing Project Task pages both land on `/404`.
- Auth/session failures and generic server failures still show their existing page-level error states.

## System-Wide Impact

- **Interaction graph:** Router unmatched handling and two workspace page branches change; backend APIs are unchanged.
- **Error propagation:** Only true missing-resource cases should convert to navigation. Other errors keep the current visible error or retry behavior.
- **State lifecycle risks:** Replacement navigation prevents users from returning directly to the bad URL via the immediate back action.
- **API surface parity:** No API contract changes and no generated API client edits.
- **Integration coverage:** Router behavior plus page-level redirects need coverage because component tests alone can miss history replacement.
- **Unchanged invariants:** Existing authenticated shell, Projects page at `/`, and non-404 error states remain unchanged.

---

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| Redirecting non-404 Project errors hides useful retry/auth feedback | Check generated API error status and only redirect true not-found responses. |
| Unmatched-route redirect loops on `/404` | Make `/404` a real route, prefer router-level redirect with replacement, and test direct rendering. |
| Route tree drifts after adding the route | Run the frontend build so `client/src/routeTree.gen.ts` is regenerated instead of hand-edited. |
| Tests overfit implementation details | Assert visible destination and replacement semantics rather than internal helper names. |
