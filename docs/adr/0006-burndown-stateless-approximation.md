# Stateless burndown approximation

The project view shows a burndown chart for the Active Sprint and for each Closed Sprint in sprint history. We compute it client-side from data the page already loads — `SprintPointSummary` for the active sprint and `ProjectSprintHistoryRead.snapshots` for closed sprints — rather than introducing persistence, a scheduler, or a new endpoint. There is no per-day task progress history in the data model; `Task` only stores its current `column_id`, `created_at`, and `updated_at`. The only historical record is `ProjectSprintTaskSnapshot`, taken once at sprint close. So a true stepped per-day burndown curve is not derivable: the active-sprint chart has two deterministic points (start total, today's remaining) plus an ideal diagonal to `planned_end_date`; a closed-sprint chart has one real point (the close snapshot) reconstructed against the start total and the same ideal diagonal.

We chose this over a `project_sprint_burndown_snapshots` table with a daily job, and over a task column-change event log, because the existing summaries already compute the numbers we need and the first useful artifact is "today's reality vs the ideal line," not a forensic per-day curve. Rejected alternatives remain viable upgrades: if scope-change detection or per-day steepness becomes important, add a snapshot table and backfill from today forward — the chart component gains points without a shape change. Known ceiling: tasks removed from the sprint mid-cycle vanish from both committed and remaining, so the chart drifts downward without a scope line to expose it. Null `story_points` follow the existing `SprintPointSummary` convention — counted in `unestimatedCount`, excluded from the points line but shown in the count line.
